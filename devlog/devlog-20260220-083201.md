# Daily Doom Devlog - 2026-02-20

## #27 - Sprite Occlusion Bug Fix

**Problem:** Enemy sprites were occasionally visible through walls, breaking 3D immersion. The `isOccludedByWall()` ray-casting method had edge cases causing sprites to "bleed through" solid barriers.

**Root Cause:** The ray-marching step size was 8 units, which could skip over wall boundaries at certain angles. Additionally, step vectors were computed by dividing by an integer step count, causing inconsistent effective step sizes.

**Fix Applied:**
- Reduced ray-march step size from 8 to 4 units for more precise wall detection
- Normalized direction vector for consistent stepping regardless of distance
- Added 16-unit buffer before sprite position to avoid false positives
- Added early-exit for zero-distance checks

**Files Modified:**
- `js/engine/renderer.js` - Improved `isOccludedByWall()` method
- `playtester/tests.js` - Added T2-15 occlusion test with 6 validation cases

**Test Results:** All 24 tests pass (T2-15 validates clear sight, wall blocking, border blocking, diagonal blocking, close sprites, and tech wall blocking).

## #28 - Enemy Visual Diversity

**Problem:** All enemy types used the same red demon sprite, making it impossible for players to distinguish between guards, imps, demons, soldiers, and other enemy types.

**Solution:** Implemented programmatic color tinting (Option 1 from the issue) to create unique visual variants for each of the 8 enemy types:
- **Imp:** Red (original) - small, fast
- **Guard:** Blue/teal - medium size
- **Soldier:** Green/brown - medium size
- **Demon:** Dark purple - large
- **Berserker:** Orange - medium, aggressive
- **Spitter:** Green - small, ranged
- **Shield Guard:** Cyan - large, defensive
- **Boss:** Gold - significantly larger (1.0x scale vs 0.6x for imps)

**Implementation:** HSL color space tinting applied to the base sprite at load time. Each enemy type gets a pre-generated canvas with its unique hue shift, saturation, and brightness multipliers. Size varies by type via per-enemy scale factors.

**Files Modified:**
- `js/engine/renderer.js` - Added `generateTintedSprites()`, `rgbToHsl()`, `hslToRgb()`, enemy tint/scale configs, updated `renderSprite()` to use type-specific sprites
- `playtester/tests.js` - Added T2-16 enemy visual diversity test

**Test Results:** All 25 tests pass.

## #29 - Enemy Movement System Fix

**Problem:** Enemies appeared static despite having AI state machines and movement code in place. All enemy types remained stationary and did not chase, patrol, or navigate the map.

**Root Cause:** A deltaTime double-conversion bug. The game loop converts deltaTime to seconds (`/ 1000`) in `game.js`, but the movement methods in both `enemy.js` and `enemy-behaviors.js` divided by 1000 again, making actual movement speed 1000x slower than intended (effectively invisible).

Additionally, `moveTowardsTarget()` and `updateFacing()` were called redundantly - both from `originalUpdate()` AND from the common section of `update()`, causing double movement for non-enhanced enemies.

**Fix Applied:**
- Fixed deltaTime usage in `moveTowardsTarget()` (`speed * deltaTime` instead of `speed * deltaTime / 1000`)
- Fixed deltaTime usage in `enhancedMovement()` in enemy-behaviors.js
- Removed duplicate `moveTowardsTarget` and `updateFacing` calls from `originalUpdate()`
- Separated enhanced AI movement (handled by `enhancedMovement()`) from basic AI movement (handled by `moveTowardsTarget()`)

**Files Modified:**
- `js/entities/enemy.js` - Fixed deltaTime bug, cleaned up duplicate movement calls
- `js/entities/enemy-behaviors.js` - Fixed deltaTime bug in enhanced movement
- `playtester/tests.js` - Added T2-17 enemy movement test

**Test Results:** All 26 tests pass. Enemies now actively move - 4/9 confirmed moving within 3 seconds with 8 distinct speed values.
